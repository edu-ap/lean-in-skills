Why design-time composability, not "let the LLM glue it together"

The position this project defends, and the alternative it answers.

The alternative position

There is a widely-held view that composability is the LLM's responsibility at runtime. Each skill advertises what it does; the model figures out which to invoke and how to chain them. Many production agent systems take this approach today, and it works well at small N.

It fails predictably as the skill catalogue grows. The failure modes are documented across the industry:

Four failure modes of runtime LLM composition at scale

1. Tool-selection lottery

When N skills have semantic overlap, the same prompt yields different tool selections across runs. The LLM is making an effectively-random choice between near-equivalent options.

a16z on the MCP workflow gap; Barry Zhang at Anthropic on reducing non-determinism with code components.

2. Context-window bloat

Every advertised skill consumes context budget. Tool definitions accumulate; the budget for actual reasoning shrinks. With 50+ skills, the model spends a measurable fraction of every turn re-deciding which tool to call rather than doing the user's task.

3. Cascading inconsistency

A skill chosen on Monday may not be chosen on Tuesday for the same task. Audit trails become non-reproducible. Production incidents become impossible to retrace because the execution path was non-deterministic.

4. Unverifiable outputs

A skill that should cite sources, return a value in range, or carry a calibration rationale isn't checked at output time. The reader catches mistakes ad-hoc, or not at all. The first signal of failure is often a downstream consumer breaking.

Two positions, side by side

Runtime LLM glue

  • Skills advertise themselves; LLM chooses dynamically
  • Composition is implicit, decided per-invocation
  • Elegant for small N; non-deterministic at large N
  • No build-time verification of output integrity
  • Context budget grows linearly with tool count
  • Audit trails non-reproducible

Design-time typed composition

  • Skills declare typed dependencies via composes:
  • Composition is explicit, decided at design time
  • Deterministic at any N
  • Typed schemas + by decide reject malformed output at save time
  • Skills organised into atomics → composites → workflows; pick the level you need
  • Audit trail is the composition graph itself

How lean-in-skills addresses each failure mode

Failure mode Design-time response
Tool-selection lottery composes: field declares dependencies explicitly; type-checking rules out alternatives that don't fit
Context-window bloat Skills organised into atomics → composites → workflows; the model picks at the level it needs and the rest collapses into a composition
Cascading inconsistency A skill's composition graph is a typed structure, not an LLM choice; same input → same execution path
Unverifiable outputs Typed schemas enforce requires_source / requires_rationale / range constraints; by decide proof rejects malformed output at save time

The boundary

This isn't a claim that runtime LLM composition is wrong everywhere. For exploratory work with a small, well-understood skill catalogue, it works. The argument is narrower:

At scale, with safety-critical outputs, the design-time approach earns its keep. The runtime camp's elegant simplicity becomes operational unreliability. The design-time camp's syntax tax becomes mechanical correctness.

The five-condition test in the "where it doesn't help" thesis doc is explicit about where each approach belongs. Most AI behaviour is perception and runtime-glue is fine. The slice we formalise is the slice where wrong-and-confident costs real money.

How verification actually runs

A worked example. A skill produces a structured output (here, a planning checklist with verdict + confidence per item). The verifier:

# Skill produces output JSON, saves it
$ /game-theory "Should we negotiate or walk away?"
# (output saved to game-analysis.json)

# Verifier vendors the schema and runs lake build
$ python3 .claude/skills/_composite/game-theory/check.py game-analysis.json
Vendored schema: projects/dev/lean-in-skills/LeanInSkills/GameAnalysis.lean
Generated instance: projects/dev/lean-in-skills/LeanInSkills/Generated/GameAnalysisInstance.lean
VERIFIED: GameAnalysisJSON satisfies structural invariants

And the failure case, where a checklist item has confidence below 0.75 but no calibration note:

$ python3 .claude/skills/_composite/game-theory/check.py malformed.json
VERIFICATION FAILED: malformed.json
  error: LeanInSkills/Generated/GameAnalysisInstance.lean:29:53:
    Tactic `decide` proved that the proposition
    gameAnalysis.wellFormed = true
    is false

The build fails. The output is not allowed to be reported as complete until the failing invariant is fixed.
